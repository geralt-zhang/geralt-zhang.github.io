## ALU

ALU（算术逻辑单元）是 CPU 中负责实际计算的核心部件，它执行两大类操作：算术运算和逻辑运算。算术运算包括加法、减法等数值计算，逻辑运算包括与、或、非、异或等布尔操作，同时还能做一些简单的状态判断，例如结果是否为 0、是否为负数。可以把 ALU 看成是 CPU 中专门负责“算”和“判断”的模块。

从硬件实现的角度看，底层电路由晶体管构成，但直接用晶体管描述运算非常复杂，因此在设计上使用逻辑门作为抽象层。常见的逻辑门有 AND、OR、NOT 和 XOR，这些基本单元可以通过组合实现各种算术功能，其中最重要的例子就是二进制加法。

在二进制加法中，XOR 门的输出正好对应加法的低位结果。例如 0+0=0、0+1=1、1+0=1。但当 1+1=10 时，XOR 只能得到 0，却无法表示进位 1。因此除了表示和的输出外，还必须额外引入一条线路表示进位，这就是加法电路设计的关键。

最简单的加法电路是半加器。半加器有两个输入 A 和 B，输出包括和与进位。和由 A XOR B 得到，进位由 A AND B 得到。半加器可以完成两个单比特的加法，但无法处理来自低位的进位。在多位加法中，每一位实际上都需要计算 A、B 以及来自前一位的进位之和，因此需要更完整的结构。

为了解决这个问题，引入了全加器。全加器有三个输入：A、B 和来自前一位的进位。它输出新的和与进位。一个全加器可以由两个半加器和一个 OR 门组合实现：先用第一个半加器计算 A 和 B，再用第二个半加器把结果与输入进位相加，最后用 OR 门合并产生的进位信号。通过这种结构，就可以正确处理连续的多位加法。

把多个全加器按位串联起来，就得到多位加法器，例如 8 位或 16 位加法器。这种结构中，每一位的进位都要传递到更高一位，称为串行进位加法器。它的缺点是进位需要逐级传播，在位数很多时会造成延迟。为提高速度，现代处理器通常采用超前进位加法器，它通过提前计算进位关系来减少等待时间，从而显著提升运算速度。

在固定长度的二进制运算中，如果最高位产生了无法存储的进位，就会发生溢出。例如 8 位二进制数 11111111 再加 1 会得到 100000000，多出来的一位无法保存。ALU 会通过状态标志位记录这种情况，同时还会记录进位、结果是否为零以及符号等信息。这些标志位会被后续指令使用，用于条件判断和程序控制。

除了加法电路之外，ALU 还包含逻辑运算单元，用来执行 AND、OR、NOT、XOR 等操作，并根据结果生成状态信息。整体来看，ALU 接收两个操作数和一个操作码，根据操作码选择具体的运算类型，输出结果以及对应的状态标志。它把底层的逻辑门组合成可以执行复杂计算的功能模块，是数字计算机完成运算任务的基础。

```bash
下面用简单的 ASCII 图把加法电路从基本逻辑门到完整加法器的演进画出来。

先从最基本的 XOR 和 AND 门开始。两个输入 A、B 同时进入 XOR 得到“和”，进入 AND 得到“进位”。

A ----\
       XOR ---- Sum
B ----/

A ----\
       AND ---- Carry
B ----/

把这两个门组合起来就是半加器。可以把它看成一个模块：

        +-------------+
A ----->|             |
        | Half Adder  |---- Sum
B ----->|             |
        |             |---- Carry
        +-------------+

半加器只能处理 A 和 B 两个输入。为了处理来自低位的进位，需要把两个半加器和一个 OR 门组合成全加器。

                 +-------------+
A -------------->| Half Adder  |---- S1 ----\
                 +-------------+             \
                                               XOR ---- Sum
                 +-------------+             /
B -------------->| Half Adder  |---- C2 ----/
        Carry-in +-------------+

C1 (来自第一个半加器的进位) ----\
                                  OR ---- Carry-out
C2 (来自第二个半加器的进位) ----/

也可以把全加器抽象成一个整体模块：

           +--------------+
A -------->|              |
B -------->|  Full Adder  |---- Sum
Carry-in ->|              |
           |              |---- Carry-out
           +--------------+

把多个全加器首尾连接，就得到多位加法器。下面是一个 4 位串行进位加法器的示意图：

        +------+    +------+    +------+    +------+
A0 ---->| FA0  |--->| FA1  |--->| FA2  |--->| FA3  |
B0 ---->|      |    |      |    |      |    |      |
Cin --->|      |    |      |    |      |    |      |
        +------+    +------+    +------+    +------+
           |           |           |           |
         Sum0        Sum1        Sum2        Sum3

每一级的 Carry-out 连接到下一级的 Carry-in，这种逐级传播的结构就是串行进位加法器。

为了加快速度，可以在结构上增加专门的进位计算逻辑，形成超前进位加法器。简化示意如下：

A,B --->[ 位加法单元 ]------\
                            \
A,B --->[ 位加法单元 ]-------> 进位计算网络 ---> 各级 Carry
                            /
A,B --->[ 位加法单元 ]------/

各位的和由本地单元计算，进位由专门的网络并行计算，而不是逐级等待，从而提高整体速度。

这些结构逐步组合在一起，就构成了 ALU 中负责加法运算的核心电路。
```

## 指令

如果一条指令只有 8 位，前 4 位是操作码，后 4 位是操作数，那么它能表达的操作种类和数据范围都非常有限。操作码只有 16 种可能，操作数也只能表示很小的数值。这种限制在实际计算中远远不够，因此现代处理器通常使用 32 位或 64 位指令，或者采用可变长指令格式。这种一条指令占用多少位的设计，称为指令长度。

在程序执行中，条件转移指令通常不会单独使用，而是先通过比较类指令（如 cmp 或 sub）改变状态标志位，再由条件转移指令根据这些标志决定是否跳转。也就是说，比较指令负责产生条件，跳转指令负责根据条件改变程序流程。

另外，ALU 本身的基础电路主要针对加法和逻辑运算，并不一定直接包含独立的乘法和除法硬件。乘除运算往往是通过多次加减、移位，或由更复杂的专用单元配合微程序实现的。从程序的角度看像是“有乘除指令”，但底层仍然建立在基本算术运算之上。

那些 mov、add、cmp、jmp 等，其实是汇编指令的例子。更准确地说，它们是某种处理器架构的汇编语言表示形式，用来对应底层的机器指令，可以这样理解——计算机真正执行的是机器指令，也就是一串二进制比特，比如：

10110000 01100001

但人类很难直接读写这些二进制，于是就用助记符来表示它们，例如：

mov al, 61h

这种用单词代替二进制指令的语言，就是汇编语言。

不同的处理器使用不同的指令集，因此对应的汇编语言也不同。指令集设计大体可以分为两种思想：CISC 和 RISC。它们的区别主要体现在指令复杂度、长度和执行方式上。

CISC（复杂指令集计算机）的特点是单条指令功能强，可以直接完成比较复杂的操作。例如在典型的 CISC 架构 x86 中，一条指令可以直接从内存读取数据、完成运算再写回内存。x86 体系主要由 Intel 推广，是个人电脑中常见的架构。

举一个简化的 x86 风格例子：

ADD [A], 5

这条指令的含义是：把内存地址 A 中的值取出来，加上 5，再写回 A。也就是说，一条指令同时包含“取数 + 运算 + 存回”。

RISC（精简指令集计算机）强调指令简单、长度固定、执行速度快。RISC 架构通常要求先把数据加载到寄存器，再进行运算，是手机和嵌入式设备中常见的代表。

同样的操作在 ARM 风格下通常要拆成多步：

```
LDR R1, [A] 	  ; 从内存读到寄存器
ADD R1, R1, #5	; 数学意思为 R1 = R1 + 5 
STR R1, [A]   	; 写回内存
```

gcc 和不同指令集之间的关系，可以理解为：gcc 负责把高级语言代码翻译成“某种处理器能执行的机器指令”，而不同的编译链决定了它要生成哪一种指令集的代码。

gcc 本身不是只服务一种 CPU 的编译器，而是一个可移植的编译框架。通过不同的后端和工具链配置，它可以为不同架构生成代码，例如 x86、ARM 或 RISC-V。所谓“不同的 gcc 编译链”，通常指的是针对不同目标平台配置好的一整套工具，包括编译器、汇编器、链接器和标准库。

- 本机 gcc：生成当前电脑 CPU 能直接运行的程序
- 交叉编译 gcc：在一台机器上生成给另一种 CPU 使用的程序

假设你在一台 x86 电脑上开发，但目标设备是 ARM 芯片，那么就需要 ARM 的交叉编译链。这个编译链会把 C 代码翻译成 ARM 指令，而不是 x86 指令。生成的汇编代码自然也会是 ARM 风格，而不是 x86 风格。

可以把流程简单看成：

C 代码 → gcc（根据目标架构选择后端）→ 目标架构的汇编 → 机器码

因此，不同的指令集和汇编语言决定了 gcc 后端如何生成代码；而不同的 gcc 编译链，本质上是为不同硬件架构准备的翻译器配置。

下面用一个很简单的 C 程序，说明同一个 .c 文件在三种不同架构（x86、ARM、RISC-V）下的编译过程和差别。

假设有这样一个文件 test.c：

```
int add(int a, int b) {
    return a + b;
}

int main() {
    int x = add(2, 3);
    return x;
}
```

第一步是一样的：不管目标架构是什么，编译器前端都会做相同的事情。它先进行预处理（展开头文件和宏），然后把 C 代码解析成语法树，再转换成一种中间表示（IR）。在这个阶段，还没有涉及具体 CPU，只是在做与平台无关的优化。

真正的分歧发生在后端阶段。编译器根据“目标架构”选择不同的代码生成器，把中间表示翻译成对应指令集的汇编。

如果目标是 x86，编译器会生成 x86 汇编，例如（示意）：

```
add:
    mov eax, edi
    add eax, esi
    ret

main:
    mov edi, 2
    mov esi, 3
    call add
```

这里使用的是 x86 的寄存器和调用约定，例如 eax、edi、esi 等。

如果目标是 ARM，同一个函数会变成 ARM 风格的汇编：

```
add:
    add r0, r0, r1
    bx lr

main:
    mov r0, #2
    mov r1, #3
    bl add
```

可以看到寄存器名称变成 r0、r1，指令形式也不同，但逻辑仍然是“把两个数相加”。

如果目标是 RISC-V，又会生成另一种汇编：

```
add:
    add a0, a0, a1
    ret

main:
    li a0, 2
    li a1, 3
    call add
```

这里使用的是 a0、a1 这样的寄存器命名和 RISC-V 指令格式。

接下来三种架构都会进入相同类型的后续步骤：汇编器把对应的汇编代码变成目标文件（机器码），链接器再把目标文件和标准库链接在一起，生成最终可执行文件。但生成的机器码格式和运行环境都与目标架构绑定，x86 的程序不能直接在 ARM 上运行，反之亦然。

汇编器将把助记符翻译成机器指令编码。不同架构的编码规则完全不同。

```
01 D8    ; add eax, ebx
C3       ; ret
```

这些字节会被放进一个 x86 目标文件（.o）。目标文件里除了这几字节机器码，还包含符号表（记录 add 这个函数名）和重定位信息。

第二步是链接。假设还有一个 main.o 调用了 add 函数。链接器会把：

add.o + main.o + 标准库

合并成一个可执行文件。在这个过程中，链接器会把 main 里“调用 add”的占位地址，替换成 add 在最终文件里的真实地址。这一步叫重定位。

链接完成后，每种架构都会得到一种不同格式的可执行二进制。

.o 是一个结构化的二进制容器，用来保存程序的各个组成部分。.o 文件通常采用 ELF 格式来组织内容。

ELF 的全称是 Executable and Linkable Format，是一种标准的二进制文件格式，用来描述目标文件、可执行文件和共享库的内部结构。它不只存机器码，而是把代码、数据和各种元信息分成多个“段”和“节”来管理。

一个典型的 .o（ELF 目标文件）大致包含这些部分：

首先是 ELF 头（ELF header）。它位于文件开头，描述这是一个什么类型的文件、针对哪种 CPU 架构、是 32 位还是 64 位等基本信息。操作系统和工具通过它判断如何解析这个文件。

然后是节表（section table），它像目录一样，列出文件里有哪些节（section）以及它们的位置和大小。常见的节包括：

```
.text 节：存放机器指令，也就是编译后的代码  
.data 节：存放已初始化的全局变量  
.bss 节：存放未初始化的全局变量（只记录大小，不占实际空间）  
.rodata 节：只读数据，比如字符串常量  
.symtab：符号表，记录函数名和变量名  
.rel.text / .rela.text：重定位信息，说明哪些地址以后需要修改
```

机器码主要存在 .text 节里，但链接器真正依赖的是符号表和重定位信息，因为它们决定了不同目标文件之间如何连接。

当多个 .o 文件被链接时，链接器会读取每个 ELF 文件的符号表，找到函数和变量的定义与引用，然后根据重定位信息修正地址，把各个节拼接成一个完整的内存布局。

最终生成的可执行文件通常仍然是 ELF 格式，但比 .o 更复杂。它除了包含节之外，还包含程序头表（program header），告诉操作系统如何把文件映射到内存，例如代码段加载到哪里、数据段加载到哪里、程序入口点在哪里。

可以把关系简单理解为：

ELF 是一种文件格式标准，.o 是采用 ELF 格式的一种目标文件，可执行文件也是采用 ELF 格式的最终产物

因此，一个 .o 文件本质上是“带有机器码和链接信息的 ELF 容器”，而不是单纯的二进制指令堆。
